# 空空码上点单（KongKong.ByteBite）— 需求清单

## 1. 产品定位

**空空码上点单**（KongKong.ByteBite）是面向烧烤店、小卖部、饭馆的轻量级扫码点单系统。顾客通过扫描店铺二维码进入 H5 页面浏览菜单并下单，商家通过 H5 管理后台管理店铺、菜单和订单。后期商家端可改造为微信小程序。

## 2. 用户角色

| 角色 | 说明 | 使用端 |
|------|------|--------|
| 顾客 | 扫码进入店铺，浏览菜单/商品，下单 | H5 网页（手机浏览器） |
| 商家/店员 | 管理店铺、菜单/商品、接单、处理订单 | H5 网页（手机端优先适配，后期改造为微信小程序） |
| 平台管理员 | 管理商家入驻、行业模板管理、平台配置 | Web 后台（PC） |

## 3. 核心业务流程

```
商家创建店铺 → 选择行业模板（可选）→ 一键应用模板商品 → 微调价格 → 生成店铺二维码 → 贴码到店铺
顾客扫码 → 进入店铺 H5 → 查看商家活动提示（如满50减5）→ 浏览菜单 → 加入购物车 → 提交订单 → 获得取货码
商家收到新订单提醒 → 接单 → 制作/备货 → 通知顾客取餐 → 顾客凭取货码或二维码取餐 → 商家核销完成订单
顾客线下付款 → 订单关闭
```

## 4. 功能模块

### 4.1 顾客端（H5 网页）

#### 4.1.1 扫码入店 [P0]

- 顾客扫描店铺二维码，进入对应店铺的 H5 点单页面
- 二维码和分享链接统一使用短链格式 `/A/{storeCode}`，例如 `http://localhost:3000/A/000001`
- 顾客端根据 6 位店铺码加载店铺菜单，不在顾客链接中暴露店铺 UUID
- 无需登录、无需手机号即可浏览菜单和下单
- 浏览器缓存店铺访问记录，下次进入自动跳转或展示最近店铺

**短链路由规则：**

| 路由 | 模块 | 说明 |
|------|------|------|
| `/A/{storeCode}` | 店铺点单页 | 当前启用，扫码/分享进入店铺 |
| `/B/{storeCode}` | 优惠活动页 | 预留，后续可展示店铺活动 |
| `/C/{storeCode}` | 会员页 | 预留，后续可接入会员能力 |
| `/store/{storeId}` | 旧链接兼容 | 需要重定向或转换到 `/A/{storeCode}` |

**扫码入店接口要求：**

| 能力 | 要求 |
|------|------|
| 按店铺码加载菜单 | `GET /api/CustomerStore/code/{storeCode}/menu` |
| 按店铺ID加载菜单 | `GET /api/CustomerStore/{storeId}/menu`，用于已知 storeId 的兼容场景 |
| 店铺不存在 | 返回业务错误 404，前端展示"店铺不存在或已下线" |
| 顾客端菜单结构 | 返回店铺信息、店铺码、分类列表、分类下商品、商品规格与可下单状态 |
| 顾客端活动信息 | 返回当前生效的商家活动摘要，至少包含活动名称、类型、满减门槛、减免金额、适用范围和有效时间，用于点单页活动提示 |

#### 4.1.1a 店铺搜索与最近店铺 [P1]

顾客除了扫码进入店铺，也需要能主动搜索店铺，解决换设备、忘记店铺码或想再次进入历史店铺的问题。

| 能力 | 要求 |
|------|------|
| 搜索入口 | 顾客端首页、我的订单页、空状态页提供"搜索店铺"入口 |
| 搜索条件 | 支持按店铺名称、店铺码、行业/分类关键词搜索；MVP 不依赖定位，后期可扩展附近店铺 |
| 搜索结果 | 展示店铺名称、门店头像/封面、行业、营业状态、活动摘要和进入按钮 |
| 可见范围 | 仅展示已启用、已发布、允许顾客访问的店铺；禁用/下线店铺不出现在搜索结果 |
| 进入店铺 | 点击结果进入 `/A/{storeCode}`，加载该店铺菜单，当前店铺购物车与其他店铺隔离 |
| 最近店铺 | 记录顾客最近访问/下单的店铺，支持从历史订单快速返回对应店铺 |
| 登录同步 | 已登录顾客的最近店铺跨设备同步；匿名顾客使用 localStorage 保存 |

**验收标准：**

- 顾客输入店铺名称或 6 位店铺码可以搜索到可访问店铺
- 点击搜索结果能进入对应店铺菜单页
- 登录后在新设备可看到历史访问/下单店铺，能快速回到原店铺

#### 4.1.2 会话恢复 [P0]

顾客再次扫码进入时，自动恢复上次的操作状态，实现无缝衔接：

| 场景 | 恢复行为 | 存储位置 |
|------|---------|---------|
| 有未提交的购物车 | 自动恢复购物车内容（商品、规格、数量、备注） | localStorage |
| 有进行中的订单 | 直接跳转到订单跟踪页，展示取货码、二维码和实时状态 | localStorage + 服务端 |
| 无购物车无进行中订单 | 进入菜单浏览页，定位到上次浏览的分类 | localStorage |
| 换了店铺扫码 | 清除旧店铺缓存，进入新店铺首页 | localStorage |

缓存策略：

| 策略 | 规则 |
|------|------|
| 购物车过期 | 同一店铺购物车缓存 24 小时有效，超时清空 |
| 订单历史上限 | 保留最近 20 条已完成订单的取货码 |
| 进行中订单 | 订单完成后 24 小时从活跃列表移至历史 |
| 换店铺 | 扫不同店铺的码时，清空当前购物车（提示用户确认），保留历史订单 |
| 缓存冲突 | 服务端数据为准，本地缓存仅做恢复提示，不覆盖服务端状态 |

#### 4.1.3 菜单浏览 [P0]

- 按分类浏览菜品/商品列表
- 每个商品展示：图片、名称、价格（起售价，多规格时显示"起"）、描述、销量
- 商品售罄/下架状态展示（灰色标记）
- 分类左侧导航 + 右侧商品列表布局

**商家营业状态对顾客端的影响：**

| 商家状态 | 顾客端展示 | 能否下单 |
|---------|-----------|---------|
| 营业中 | 正常展示菜单，可浏览可下单 | ✅ 可以 |
| 休息中 | 顶部横幅提示"商家休息中，暂不接单"，菜单仍可浏览但按钮置灰 | ❌ 不可以 |
| 营业时间外 | 顶部横幅提示"商家尚未营业（营业时间 XX:XX-XX:XX）"，可浏览菜单 | ❌ 不可以 |

- 休息中/营业时间外，顾客仍可浏览菜单和添加购物车，但"去结算"按钮置灰
- 购物车内容保留，商家恢复营业后顾客可直接下单
- 商家切换营业状态时，已连接的顾客端通过 SignalR 实时更新页面状态

#### 4.1.3b 商家活动提示 [P0]

顾客进入店铺点单页后，需要在点单过程中持续感知商家正在进行的优惠活动，尤其是满减活动，帮助顾客判断当前订单是否满足优惠条件。

| 展示位置 | 展示内容 | 交互要求 |
|---------|---------|---------|
| 店铺顶部/菜单页活动条 | 展示当前生效的商家活动，如"满50减5"、"满100减15" | 多个活动时优先展示最容易达到或优惠力度最大的活动，支持点击展开全部活动 |
| 商品列表/购物车浮层 | 展示当前购物车距离满减门槛的差额，如"再买 ¥12 可享满50减5" | 购物车金额变化后实时更新提示 |
| 下单确认页 | 展示已满足/未满足的活动、优惠金额和最终应付金额 | 已满足活动自动应用或推荐最优活动，提交前以后端重新计算结果为准 |
| 活动不可用场景 | 展示不可用原因，如未到活动时间、指定分类/商品不满足、商家休息中暂不可下单 | 不可用提示不阻塞菜单浏览 |

**活动提示规则：**

| 规则 | 说明 |
|------|------|
| 活动来源 | 使用商家配置且处于生效时间内的优惠活动，MVP 优先支持满减活动 |
| 展示优先级 | 已满足的活动优先展示；未满足时展示距离最近门槛的满减活动 |
| 多档满减 | 支持"满50减5、满100减15"等多档提示，购物车金额达到更高档时自动切换到更优档 |
| 适用范围 | 全店活动直接按购物车合计计算；分类/商品活动仅按符合范围的商品金额计算 |
| 数据一致性 | 前端活动提示仅作引导，订单提交时由后端重新匹配活动并锁定优惠金额 |

#### 4.1.4 商品搜索 [P1]

- 关键词搜索菜品/商品
- 搜索结果高亮匹配文字

#### 4.1.5 购物车 [P0]

- 添加/删除/修改商品数量
- 选择商品规格（如：大份/小份，不同规格不同价格）
- 单品备注（少辣、不加香菜等）
- 实时计算合计金额
- 购物车浮动按钮（显示商品数量角标）
- 购物车区域展示商家活动提示，如"满50减5"、"已减 ¥5"、"再买 ¥12 可减 ¥5"
- 清空购物车

#### 4.1.6 下单 [P0]

- 确认订单信息（商品列表、规格、数量、备注、合计）
- 选择就餐方式：堂食 / 打包 / 外卖
- 堂食时填写桌号（手动输入，非扫码绑定）
- 外卖时填写配送地址和联系电话
- 提交订单（先做后付，无需在线支付）
- 下单成功后生成取货码：数据库以 `int` 存储原始码值，页面以 6 位 Base36（0-9、A-Z）大写字符串展示
- 取货码页面展示，支持截图保存，并同步展示二维码，便于商家使用手机扫码快速核销

**优惠计算流程（下单确认页）：**

```
购物车合计 → 系统自动匹配可用优惠活动
         → 展示优惠信息：
             ├── 满减：满50减5（当前合计¥66，已满足）
             ├── 折扣：烧烤类8折（当前合计中烧烤类¥50，折后¥40）
             └── 优惠不叠加提示（商家配置）
         → 顾客选择优惠方式（若可叠加则自动应用最优组合）
         → 实时更新：
             ├── 商品合计：¥66.00
             ├── 优惠金额：-¥5.00（满减）
             ├── 应付金额：¥61.00
             └── 优惠明细展开/收起
         → 提交订单
```

**下单确认页展示元素：**

| 区域 | 内容 |
|------|------|
| 商品列表 | 每个商品的名称、规格、数量、单价、小计 |
| 优惠区域 | 可用满减/折扣列表，自动推荐最优，顾客可手动切换 |
| 优惠明细 | 优惠类型、优惠金额、满足条件说明 |
| 合计区域 | 商品合计、优惠减免、应付金额 |
| 就餐方式 | 堂食（填桌号）/ 打包 / 外卖（填地址） |
| 备注 | 整单备注（如：不要葱） |

**优惠匹配规则：**

| 规则 | 说明 |
|------|------|
| 自动匹配 | 系统自动计算所有可用优惠，推荐最优组合 |
| 顾客可切换 | 顾客可在多个可用优惠中手动选择 |
| 不满足时提示 | "再买 ¥XX 可享受满减优惠"，引导凑单 |
| 优惠互斥 | 商家配置是否允许满减与折扣叠加 |
| 价格以后端为准 | 前端展示为参考，提交时后端重新计算并锁定 |

#### 4.1.7 取货码与订单跟踪 [P0]

- 下单成功后展示 6 位 Base36 取货码和二维码
- 取货码、二维码核销链接、订单 ID、店铺 ID 存储在浏览器缓存（localStorage）
- 再次进入时可通过取货码或订单 ID 查看订单状态
- 订单状态：待接单 → 制作中 → 待取餐 → 已完成
- 订单状态变更时页面实时更新（SignalR）

**顾客取消订单：**

| 订单状态 | 能否取消 | 说明 |
|---------|---------|------|
| 待接单 | ✅ 可以 | 顾客可随时取消，订单直接标记"已取消" |
| 制作中 | ❌ 不可以 | 商家已开始制作，顾客需联系商家协商 |
| 待取餐 | ❌ 不可以 | 商家已完成制作，顾客需到店取餐 |
| 已完成 | ❌ 不可以 | 订单已关闭 |

- 顾客在"待接单"状态下点击"取消订单"→ 二次确认 → 订单标记"已取消"
- 取消后取货码失效，从活跃订单列表移除
- 商家端收到订单取消通知（SignalR）

**商家拒单：**

- 商家在"待接单"状态下可拒单，需填写拒单原因
- 顾客端实时收到拒单通知，展示拒单原因
- 取货码失效，订单标记"已取消"

**退款流程（后期）：**

- MVP 阶段为"先做后付"，无需退款流程
- 后期接入在线支付后，取消/拒单自动退款

#### 4.1.8 订单历史 [P1]

- 浏览器缓存记录历史取货码列表
- 点击取货码查看对应订单详情
- 再来一单（一键加入购物车）
- 已登录顾客的"我的订单"按店铺分组展示，避免多店铺订单混在一起难以识别
- 订单分组展示店铺名称、店铺码、最近一笔订单状态和"进入店铺"按钮
- 历史订单详情页保留"进入该店铺"能力，快速回到之前下单的店铺菜单
- 匿名订单按 deviceId 查询；登录后按 customerId 查询，并合并本机匿名订单

#### 4.1.9 顾客注册/登录 [P0]

顾客可选择性注册账号，注册后将浏览器缓存中的匿名数据合并到账号中，实现跨设备数据同步。

**注册/登录方式：**

| 方式 | 要求 | 说明 |
|------|------|------|
| 手机号注册/登录 | 手机号 + 验证码；可选设置密码 | 适合顾客快速注册，手机号唯一 |
| 账号注册/登录 | 账号名 + 密码 | 不强制手机号，账号名需全局唯一，避免与管理员/商家/顾客已有账号冲突 |
| 统一登录 | 登录入口支持手机号或账号名 | 后端自动识别顾客身份，登录后回到当前店铺或我的订单 |
| 微信授权登录 | 后期支持 | 小程序版使用，可绑定已有手机号或账号 |

**顾客端直接注册入口：**

| 入口 | 行为 |
|------|------|
| 我的页面 | 未登录时展示"登录/注册"，登录后展示账号信息、最近店铺、我的订单 |
| 我的订单页 | 未登录时允许查看本机匿名订单，同时提示注册防止跨设备丢失 |
| 下单成功页/订单详情页 | 展示"保存到账号"入口，注册后自动合并当前订单 |
| 店铺搜索页 | 未登录可搜索，登录后同步最近访问店铺 |

**注册流程与数据合并：**

```
顾客扫码点单（匿名）→ 浏览器缓存积累数据（订单、购物车、访问记录）
                   → 选择注册
                   → 选择手机号注册或账号注册
                   → 系统检测到本地缓存数据
                   → 提示"发现您之前在以下店铺有订单记录，是否合并到新账号？"
                   → 展示待合并数据摘要：
                      ├── XX烧烤店：3笔订单（1笔进行中、2笔已完成）
                      ├── XX便利店：1笔订单（已完成）
                      └── XX面馆：购物车中有2件商品
                  → 用户确认合并
                  → 系统执行合并：
                      1. 匿名订单 → 关联到注册用户
                      2. 浏览器缓存购物车 → 同步到用户账号
                      3. 访问记录 → 同步到用户账号
                  → 合并完成，清除本地缓存
                  → 后续登录自动同步数据
```

**多店铺数据合并规则：**

| 数据类型 | 合并策略 | 说明 |
|----------|---------|------|
| 进行中订单 | 全部合并 | 我的订单按店铺分组展示，保留取货码和二维码 |
| 已完成订单 | 全部合并 | 合并到账号订单历史，先按店铺分组，再按时间倒序展示 |
| 购物车 | 按店铺分别合并 | 保留各店铺购物车内容，不跨店混合 |
| 访问记录 | 合并去重 | 保留最近访问和最近下单的店铺列表，支持一键进入店铺 |
| 取货码 | 全部合并 | 合并到账号取货码列表 |

**数据合并冲突处理：**

| 场景 | 处理方式 |
|------|---------|
| 本地缓存与账号已有相同店铺订单 | 以服务端数据为准，本地数据补充缺失的订单 |
| 本地购物车与账号购物车冲突 | 合并去重，同一商品取数量较大值 |
| 同一订单在本地和账号都存在 | 关联到账号，不重复创建 |

**注册后数据同步：**

| 场景 | 行为 |
|------|------|
| 已登录用户扫码 | 订单直接关联到账号，不存 localStorage |
| 已登录用户换设备 | 登录后自动拉取账号数据，无需手动合并 |
| 已登录用户进入不同店铺 | 菜单、购物车和活动按当前店铺展示；我的订单仍可看到该账号下所有店铺订单 |
| 已登录用户退出登录 | 降级为匿名模式，数据存 localStorage |
| 匿名用户在已注册设备注册 | 自动触发合并流程 |

**顾客账号信息：**

| 字段 | 必填 | 说明 |
|------|------|------|
| 手机号 | 手机号注册必填 | 作为手机号登录标识，唯一 |
| 账号名 | 账号注册必填 | 作为账号登录标识，唯一，建议 4-20 位字母/数字/下划线 |
| 密码哈希 | 账号注册必填 | 手机号验证码注册可后续补充密码 |
| 昵称 | 否 | 手机号注册默认手机号后四位，账号注册默认账号名 |
| 头像 | 否 | 默认系统头像 |

#### 4.1.10 顾客退出登录 [P1]

已登录的顾客可主动退出登录，退出后降级为匿名模式继续使用。

**退出登录行为：**

| 行为 | 说明 |
|------|------|
| 清除登录态 | 前端清除 Token/用户信息缓存 |
| 降级为匿名模式 | 退出后仍可扫码点单，数据存 localStorage |
| 进行中订单保留 | 退出登录不影响已有订单，取货码仍可查看 |
| 购物车保留 | 退出登录后购物车内容不清空 |
| API 接口 | `POST /api/customers/logout`，服务端清除 Token 即可 |

#### 4.1.11 订单消息与商家会话 [P1]

顾客在已下单订单中可以向店铺老板发起消息，用于催单、修改备注、询问取餐、协商取消等场景。会话以订单为上下文，消息需实时送达并可回看。

| 能力 | 要求 |
|------|------|
| 发起入口 | 订单详情页、订单跟踪页提供"联系商家"入口；未下单前不开放订单会话 |
| 会话范围 | 会话绑定 `orderId + storeId + customerId/deviceId`，同一订单只创建一个会话 |
| 消息类型 | MVP 支持文本消息；后期可扩展图片、语音和快捷短语 |
| 实时通信 | 顾客发送消息后，商家端通过 SignalR 实时收到提醒；商家回复后顾客端实时收到 |
| 离线补偿 | 任一端离线时消息入库，重新进入订单/会话页后拉取未读消息 |
| 未读提示 | 顾客端我的订单、订单详情展示未读回复数量；商家端订单列表/会话列表展示未读顾客消息 |
| 权限边界 | 只有该订单所属顾客（或同 deviceId 匿名顾客）和该店铺商家可查看/发送消息 |
| 订单联动 | 顾客在订单会话中可查看订单状态、取货码和商家回复，不影响原订单状态流转 |

**顾客端验收标准：**

- 顾客从订单详情点击"联系商家"可以进入会话
- 发送消息后商家端无需刷新即可收到
- 商家回复后顾客端无需刷新即可看到
- 换设备登录后可以看到该账号下历史订单会话记录

#### 4.2.3 分类管理 [P0]

商家可完全自定义商品分类，不受行业模板限制，自由创建符合自身经营特色的分类体系。

**分类属性：**

| 属性 | 说明 | 示例 |

| 分类类型 | 普通分类 / 特殊分类 | 见下方分类类型说明 |
| 排序权重 | 数值越大越靠前 | 热销=1000，进店福利=900 |
| 显示状态 | 显示 / 隐藏 | 临时下架某分类时使用 |

**分类类型：**

| 类型 | 说明 | 顾客端展示 | 商品来源 |
|------|------|-----------|---------|
| 普通分类 | 商家手动归类的商品 | 正常展示，按排序权重排列 | 商家手动将商品归入 |
| 热销 | 自动聚合销量 Top N 的商品 | 置顶展示，标签"🔥 热销" | 系统自动按销量计算，商家可设置 Top 数量（默认 10） |

- 编辑分类：修改名称、图标、类型、排序
- 删除分类：分类下有商品时需先移走商品或选择一并删除

**分类与商品关系：**

| 规则 | 说明 |

| 套餐分类下的商品为虚拟商品 | 套餐商品由多个普通商品组合而成，有独立的组合价格 |
| 分类为空时顾客端不展示 | 某分类下无上架商品时，顾客端自动隐藏该分类 |

| 实时提醒 | 顾客发送消息后商家端通过 SignalR 实时弹出提醒，并更新会话列表未读数 |
| 订单上下文 | 会话详情顶部固定展示关联订单摘要：取货码、订单状态、下单时间、金额、就餐方式 |
| 打开订单 | 商家在会话中可一键打开对应订单详情，进行接单、备餐、核销等操作 |
| 回复消息 | 支持商家发送文本回复，顾客端实时收到；回复内容保存到消息历史 |
| 权限校验 | 商家只能查看和回复自己店铺下的会话，不能跨店查看顾客消息 |
| 已读状态 | 商家打开会话后标记顾客消息已读，顾客端可看到回复但 MVP 不强制展示已读回执 |

**商家端验收标准：**

- 顾客从订单发消息后，商家端会话列表实时出现该消息
- 商家打开会话时能看到对应订单摘要，并能一键进入订单详情
- 商家回复后，顾客端会话页实时显示回复内容
- A 店商家不能看到或回复 B 店订单会话

### 4.3 系统级功能

#### 4.3.1 SignalR 实时推送 [P0]

SignalR 是本系统实时通信的核心，顾客端和商家端均需接入，确保订单状态变更和订单会话消息即时触达。

**推送场景与消息定义：**

| 编号 | 推送方向 | 触发事件 | 消息内容 | 接收端行为 |
|------|---------|---------|---------|-----------|
| S1 | 顾客 → 商家 | 顾客下单 | 订单摘要（订单号、商品数、合计、就餐方式） | 商家端弹窗提醒 + 声音 + 震动，订单列表刷新 |
| S2 | 商家 → 顾客 | 商家接单 | 接单确认（预计等待时间，可选） | 顾客端订单状态更新为"制作中"，页面提示 |
| S3 | 商家 → 顾客 | 商家拒单 | 拒单原因 | 顾客端订单状态更新为"已取消"，展示拒单原因 |
| S4 | 商家 → 顾客 | 备餐完毕/待取餐 | 取餐提醒 + 取货码 + 二维码 | 顾客端订单状态更新为"待取餐"，高亮取货码并展示二维码 |
| S5 | 商家 → 顾客 | 外卖已送出 | 送出提醒 + 预计送达时间（可选） | 顾客端订单状态更新为"配送中"（预留） |
| S6 | 商家 → 顾客 | 订单完成 | 完成确认 | 顾客端订单状态更新为"已完成" |
| S7 | 系统 → 商家 | 订单超时未接 | 超时提醒（订单号 + 等待时长） | 商家端提醒该订单已等待超过 N 分钟 |
| S8 | 顾客 → 商家 | 顾客发送订单消息 | 消息内容 + 会话 ID + 订单摘要 | 商家端会话列表实时更新并提示未读 |
| S9 | 商家 → 顾客 | 商家回复订单消息 | 回复内容 + 会话 ID + 订单状态摘要 | 顾客端订单会话实时显示回复 |

**连接管理：**

| 环节 | 顾客端 | 商家端 |
|------|--------|--------|
| 连接时机 | 进入店铺页面时自动建立连接 | 登录后自动建立连接，全程保持 |
| 连接标识 | 匿名或登录连接，通过订单 ID/会话 ID 订阅特定订单消息 | 认证连接，通过店铺 ID 订阅店铺所有订单和会话消息 |
| 断线重连 | 自动重连（指数退避，最多 5 次） | 自动重连（指数退避，最多 10 次） |
| 重连后 | 主动拉取订阅订单的最新状态，补齐断线期间变更 | 主动拉取店铺订单列表最新状态 |
| 离开页面 | 断开连接，释放服务端资源 | 保持连接（多标签页共享连接） |

**Hub 设计：**

```
SignalR Hubs
├── OrderHub（顾客端）
│   ├── 订阅订单：SubscribeOrder(orderId)
│   ├── 取消订阅：UnsubscribeOrder(orderId)
│   └── 服务端推送方法：
│       ├── OrderStatusChanged(orderId, status, data)
│       └── OrderCancelled(orderId, reason)
│
├── ConversationHub（订单会话）
│   ├── 订阅会话：SubscribeConversation(conversationId)
│   ├── 取消订阅：UnsubscribeConversation(conversationId)
│   └── 服务端推送方法：
│       ├── CustomerMessageReceived(conversationId, orderSummary, message)
│       └── MerchantMessageReceived(conversationId, orderSummary, message)
│
└── StoreHub（商家端）
    ├── 订阅店铺：SubscribeStore(storeId)
    ├── 取消订阅：UnsubscribeStore(storeId)
    └── 服务端推送方法：
        ├── NewOrderReceived(orderId, summary)
        ├── OrderTimeoutWarning(orderId, waitingMinutes)
        └── ConversationUnreadChanged(conversationId, orderId, unreadCount)
```

**降级策略：**

| 场景 | 降级行为 |
|------|---------|
| SignalR 连接失败 | 降级为 HTTP 轮询（每 5 秒查询一次订单状态） |
| 会话 SignalR 连接失败 | 消息发送仍走 HTTP 接口入库，页面定时拉取未读消息和会话列表 |
| 轮询也失败 | 页面展示"网络异常"提示，提供手动刷新按钮 |
| 商家端长时间无响应 | 顾客端展示"商家可能未在线，请耐心等待或到店咨询" |

**订单会话数据要求：**

| 对象 | 关键字段 |
|------|---------|
| Conversation | `id`、`orderId`、`storeId`、`customerId`、`deviceId`、`lastMessageAt`、`customerUnreadCount`、`merchantUnreadCount`、`createdAt` |
| ConversationMessage | `id`、`conversationId`、`senderType(customer/merchant/system)`、`senderId`、`content`、`messageType(text)`、`readAt`、`createdAt` |

| 规则 | 要求 |
|------|------|
| 唯一性 | 同一订单只允许创建一个会话，避免一个订单出现多个聊天入口 |
| 归属 | 会话必须绑定订单和店铺，消息权限跟随订单归属和店铺归属校验 |
| 查询 | 顾客按 `customerId/deviceId` 查询自己的会话；商家按 `storeId` 查询店铺会话 |
| 保留 | 订单完成后会话仍可回看，是否允许继续发送消息可由店铺配置或默认保留 24 小时 |

**从商家引入流程：**

```
管理员选择"从商家创建模板"
  → 选择商家（支持搜索/按行业筛选）
  → 预览商家当前分类和商品
  → 选择引入范围：
      ├── 全部引入（一键提取该商家所有分类和商品）
      └── 部分引入（勾选需要的分类和商品）
  → 设置模板信息（名称、行业分类、描述）
  → 系统自动执行：
      1. 商家分类 → 模板分类（保留分类类型和排序）
      2. 商家商品 → 模板商品（价格转为"参考价格"，图片复制到模板资源目录）
      3. 商家商品规格 → 模板规格（加价保留）
```

**多商家组合流程：**

```
管理员选择"组合创建模板"
  → 设置模板基础信息（名称、行业分类、描述）
  → 进入组合编辑界面：
      左侧：商家列表（可搜索/筛选）
      右侧：当前模板内容（分类 + 商品）
  → 从商家A选取：
      选择商家A → 展示其分类和商品
      → 勾选需要的分类/商品 → 添加到模板
  → 从商家B选取：
      选择商家B → 展示其分类和商品
      → 勾选需要的分类/商品 → 追加到模板
  → ... 可继续从更多商家选取
  → 管理员调整：
      ├── 合并同名分类（如：商家A和商家B都有"烧烤"分类 → 合并为一个分类）
      ├── 去重同名商品（如：两个商家都有"烤羊肉串" → 保留一个或两个都保留为不同参考价）
      ├── 调整排序
      └── 修改参考价格
  → 保存模板
```

**从商家引入的规则：**

| 规则 | 说明 |
|------|------|
| 价格转为参考价 | 商家实际售价转为模板"参考价格"，新商家应用后可自行调整 |
| 图片复制 | 商家商品图片复制到模板资源目录，不引用原商家图片（避免商家删图影响模板） |
| 不影响原商家 | 引入操作只读取商家数据，不修改原商家的任何内容 |
| 商家无感知 | 商家不知道自己的商品被引入模板（无需商家授权） |
| 同名分类合并 | 多商家组合时，同名分类自动合并，商品归入同一分类 |
| 同名商品处理 | 多商家组合时，同名商品默认保留第一个引入的，管理员可选择保留哪个或都保留 |
| 套餐可引入 | 商家的套餐商品也可引入模板，子商品引用自动映射为模板商品引用 |
| 引入后独立 | 引入后的模板数据与原商家完全独立，后续商家修改不影响模板 |

**关键规则：**

| 规则 | 说明 |
|------|------|
| 模板商品默认下架 | 应用后商品状态为"下架"，商家确认价格后才上架，避免错误价格暴露给顾客 |
| 价格仅供参考 | 模板价格为市场参考价，商家必须自行确认后才能上架 |
| 图片可替换 | 模板图片为通用示意图，商家应用后可替换为自己的实拍图 |
| 可跳过模板 | 商家可选择"自定义"跳过模板，从零配置 |
| 可部分应用 | 商家可只应用模板中的部分分类和商品，不需要全部采用 |
| 模板不锁定 | 应用模板后，商家可自由增删改分类和商品，不受模板约束 |
| 可重复应用 | 商家可在已有商品的基础上，追加应用其他模板（合并模式） |

#### 4.3.6 统一登录与 Token [P0]

系统提供统一登录能力，用户不需要先选择角色，输入账号后由后端自动识别登录身份。

**统一登录规则：**

| 账号格式 | 识别策略 | 登录成功后跳转 |
|----------|----------|----------------|
| 非 11 位手机号 | 依次匹配管理员账号 → 顾客账号名 | 管理后台或顾客端 |
| 11 位手机号 | 依次尝试商家 → 顾客 | 商家端或顾客端 |
| 均未匹配 | 返回 401 | 保持登录页并提示 |

**账号唯一性：**

- 顾客账号名需与管理员账号、商家账号和其他顾客账号保持唯一，避免统一登录歧义
- 手机号作为登录名时同一角色内唯一；若同一手机号同时是商家和顾客，统一登录优先进入商家端，顾客端可提供"切换为顾客身份"

**登录状态限制：**

| 角色 | 禁止登录状态 | 说明 |
|------|--------------|------|
| 商家 | 待审核、已禁用、已封禁 | 提示账号状态异常 |
| 管理员 | 已禁用 | 禁止进入后台 |
| 顾客 | 无特殊限制 | 可匿名使用，也可登录同步数据 |

**Token 规则：**

| 项目 | 要求 |
|------|------|
| 生成方式 | MVP 阶段使用服务端生成的随机 Token |
| 存储位置 | 服务端实体 Token 字段 + 前端 localStorage |
| 请求传输 | `Authorization: Bearer {token}` |
| 退出登录 | 前端清除 Token，服务端清空对应实体 Token |
| 兼容接口 | 保留商家、管理员、顾客各自登录接口，统一登录作为主入口 |

#### 4.3.7 设备标识与匿名顾客 [P0]

顾客端无需注册即可使用，但系统需要稳定识别同一浏览器设备，支撑匿名下单、购物车恢复和后续数据合并。

| 能力 | 要求 |

| 设备标识生成 | 首次访问时生成 deviceId，并持久化到 localStorage |

| 下单关联 | 匿名订单记录 deviceId，注册用户订单记录 customerId |

| 隐私边界 | deviceId 仅作为业务恢复标识，不作为真实身份认证凭证 |

#### 4.3.8 统一 API 响应与错误处理 [P0]

所有后端接口统一返回 `{ code, message, data, detail }` 结构，前端通过请求拦截器统一解包和提示错误。

## 5. 取货码机制

### 5.1 设计说明

取货码是本系统的核心交互凭证，替代传统手机号验证流程。取货码需要同时满足顾客容易出示、商家容易核销、数据库容易索引和后续可扩展扫码核销的要求。

| 环节 | 说明 |
|------|------|
| 生成时机 | 顾客下单成功后自动生成 |
| 数据库存储 | 订单表存储有符号 32 位整数码值，例如 `pickupCodeValue int`；不以展示字符串作为主存储字段 |
| 展示格式 | 页面统一显示 6 位 Base36 大写码，字符集为 `0-9`、`A-Z`，不足 6 位左侧补 `0`，例如 `00000A`、`00F3K9` |
| 编码规则 | 与店铺码使用同一 Base36 编码能力；显示值 = `Base36(pickupCodeValue).PadLeft(6, '0')` |
| 码值范围 | `1` 到 `2,147,483,647`（有符号 `int` 上限，Base36 为 `ZIK0ZJ`），`0` 可保留为无效/未生成状态；如未来要覆盖完整 6 位 Base36 空间，需要改为 `bigint` |
| 唯一性 | 同一店铺内，进行中订单的取货码不得重复；已完成/已取消/已拒单订单的码值可在安全窗口后复用 |
| 存储方式 | 浏览器 localStorage 缓存订单 ID、店铺 ID、展示取货码和订单状态，用于会话恢复；服务端数据始终为准 |
| 展示方式 | 订单详情页大字展示 6 位取货码，同时展示二维码，支持截图保存 |
| 核销方式 | 商家端支持手动输入 6 位取货码核销，也支持手机扫码二维码快速核销 |
| 有效期 | 订单完成后保留 24 小时用于顾客回看，之后从活跃列表移除；码值是否可复用以服务端安全窗口为准 |

### 5.2 取货码生成规则

| 规则 | 要求 |
|------|------|
| 生成来源 | 后端在订单创建事务内生成，不由前端生成 |
| 生成策略 | 可采用随机整数或按店铺维度递增/扰动生成；必须在数据库层或事务内保证同店铺进行中订单唯一 |
| 取值边界 | 生成码值必须落在 `1` 到 `2,147,483,647` 范围内，避免超出数据库 `int` 存储上限 |
| 冲突处理 | 发现同店铺进行中订单码值冲突时重新生成，达到重试上限后返回业务错误并记录日志 |
| 排除状态 | `completed`、`cancelled`、`rejected` 不参与进行中唯一性校验，但短时间内复用需避免影响顾客取餐体验 |
| 展示转换 | API 返回给前端时提供 `pickupCode` 字符串；服务端内部和数据库以 `pickupCodeValue` 整数为准 |
| 排序和查询 | 商家手动输入取货码时，后端将 Base36 字符串解码为整数，再按 `storeId + pickupCodeValue` 查询 |

### 5.3 顾客端取货码与二维码流程

```
下单成功
  → 后端生成 pickupCodeValue
  → API 返回 6 位 pickupCode 展示值和二维码核销链接
  → 顾客端订单详情页展示：大字取货码 + 二维码 + 当前订单状态
  → 取货码信息写入 localStorage 作为会话恢复线索
  → 再次进入 → 自动展示未完成订单的取货码列表
  → 点击取货码 → 查看订单详情与状态
```

**顾客端展示要求：**

| 区域 | 要求 |
|------|------|
| 大字取货码 | 6 位等宽展示，字符间距清晰，适合远距离出示 |
| 二维码 | 与取货码同屏展示；支持点击放大，避免商家扫码时识别面积过小 |
| 辅助信息 | 展示门店名、订单状态、下单时间、就餐方式，帮助商家人工核对 |
| 离线/弱网 | 已加载过的订单详情可继续展示取货码和二维码；核销仍必须由商家端联网完成 |

### 5.4 二维码核销规则

二维码不应只存裸取货码，建议存放核销链接，便于商家用手机扫码后直接进入核销页面：

```
/merchant/verify?storeCode={storeCode}&pickupCode={pickupCode}&orderId={orderId}
```

| 项目 | 要求 |
|------|------|
| 二维码内容 | 使用核销链接，至少包含店铺码、展示取货码；可包含订单 ID 作为精确定位线索 |
| 权限校验 | 商家扫码后必须登录；后端校验该商家是否拥有该店铺，不能跨店核销 |
| 状态校验 | 仅 `ready` 待取餐状态可直接核销完成；`pending/accepted/preparing` 扫码时提示订单尚未可取；`completed/cancelled/rejected` 提示已失效 |
| 二次确认 | 扫码命中订单后展示订单摘要，商家点击确认后才标记完成，避免误扫即核销 |
| 手动兜底 | 二维码无法识别时，商家仍可输入 6 位取货码查询并核销 |
| 核销结果 | 核销成功后订单状态变为 `completed`，记录完成时间和操作人，并通过 SignalR 通知顾客端 |
| 安全边界 | 二维码不包含顾客手机号、地址等隐私信息；核销动作必须走服务端鉴权接口 |

### 5.5 商家端取货码核销流程

```
订单标记待取餐
  → 顾客出示取货码或二维码
  → 商家手动输入取货码，或使用手机扫描二维码
  → 系统匹配当前店铺订单并展示订单摘要
  → 商家确认核销
  → 订单标记已完成
```

### 5.6 验收标准

| 场景 | 验收标准 |
|------|---------|
| 下单生成 | 新订单数据库存在整数取货码值，API 返回 6 位 Base36 展示码 |
| 位数补齐 | 小整数码值显示时必须左侧补 0，始终 6 位 |
| 边界安全 | 不生成超过 `int` 上限的码值，边界值 `2,147,483,647` 显示为 `ZIK0ZJ` |
| 唯一性 | 同一店铺多个进行中订单的展示取货码不重复 |
| 顾客展示 | 订单详情页同时显示大字取货码和二维码 |
| 二维码扫码 | 商家扫码后进入核销确认页，确认后订单完成 |
| 权限隔离 | A 店商家不能核销 B 店订单 |
| 状态限制 | 非待取餐订单扫码不直接完成，页面给出明确状态提示 |
| 回归兼容 | 历史字符串取货码如已存在，需要迁移或兼容展示，避免旧订单详情打不开 |

## 6. 商品规格设计

### 6.1 数据模型

```
商品（Product）
├── 基础价格（默认规格价格）
├── 规格组（SpecGroup）
│   ├── 规格组名称（如：份量）
│   └── 规格项（SpecOption）
│       ├── 规格名称（如：小份）
│       ├── 加价（如：+0 / +10）
│       └── 库存（可选）
└── 规格组（SpecGroup）
    ├── 规格组名称（如：辣度）
    └── 规格项（SpecOption）
        ├── 规格名称（如：微辣）
        ├── 加价（如：+0）
        └── 库存（可选）
```

### 6.2 价格计算

- 商品最终价格 = 基础价格 + 各规格组选中项的加价之和
- 顾客端显示"起售价"（基础价格），选择规格后实时更新价格
- 购物车和订单中记录每个商品的具体规格和最终价格

### 6.3 示例

| 商品 | 基础价格 | 规格组 | 规格项 | 加价 | 最终价格 |
|------|---------|--------|--------|------|---------|
| 烤羊肉串 | ¥15 | 份量 | 小份 | +0 | ¥15 |
| | | | 大份 | +10 | ¥25 |
| | | 辣度 | 不辣 | +0 | — |
| | | | 微辣 | +0 | — |
| | | | 特辣 | +0 | — |
| 可乐 | ¥3 | 规格 | 听装 | +0 | ¥3 |
| | | | 瓶装 | +2 | ¥5 |

### 6.4 套餐商品数据模型

套餐是一种特殊的虚拟商品，由多个普通商品组合而成，拥有独立的组合价格。

```
套餐商品（ComboProduct）
├── 基础信息
│   ├── 套餐名称（如：2人超值套餐）
│   ├── 套餐价格（独立定价，不等于子商品价格之和）
│   ├── 套餐描述（如：含烤羊肉串2份+烤鸡翅4份+啤酒2瓶）
│   ├── 套餐图片
│   ├── 所属分类（套餐类型分类）
│   ├── 排序权重
│   ├── 商品状态：上架 / 下架 / 售罄
│   ├── 起购数量（默认 1）
│   └── 可选规格组（如：辣度选择，套餐整体适用）
│
├── 包含商品列表（ComboItem）
│   └── 子商品引用
│       ├── 引用的普通商品 ID
│       ├── 数量（如：2 份烤羊肉串）
│       ├── 默认规格（如：大份）
│       ├── 是否允许顾客替换规格（可选）
│       └── 子商品备注（如：默认微辣）
│
└── 套餐优惠信息
    ├── 原价总和（各子商品按默认规格的价格之和）
    ├── 套餐价格（独立定价，通常低于原价总和）
    └── 优惠金额（原价总和 - 套餐价格）
```

**套餐价格计算规则：**

| 规则 | 说明 |
|------|------|
| 套餐独立定价 | 套餐价格由商家手动设置，不自动计算 |
| 顾客端展示 | 显示"原价 ¥120 · 套餐价 ¥88 · 省 ¥32" |
| 子商品规格替换 | 若顾客替换了子商品规格，套餐价格不变（规格差价另计） |
| 套餐与满减 | 套餐商品可参与满减活动，套餐价格计入满减门槛 |
| 套餐内商品售罄 | 任一子商品售罄，整个套餐自动标记售罄 |
| 套餐内商品下架 | 任一子商品下架，整个套餐自动下架 |

**套餐示例：**

| 套餐 | 套餐价 | 包含商品 | 原价总和 | 优惠金额 |
|------|--------|---------|---------|---------|
| 2人超值套餐 | ¥88 | 烤羊肉串×2(大份) + 烤鸡翅×4 + 烤茄子×1 + 雪花啤酒×2(瓶装) | ¥120 | ¥32 |
| 4人畅吃套餐 | ¥168 | 烤羊肉串×4(大份) + 烤鱼×1(整条) + 烤鸡翅×8 + 烤韭菜×2 + 雪花啤酒×4(瓶装) | ¥236 | ¥68 |

### 8.2 目录结构

```
uploads/

│   └── {storeId}/                    # 每个商家一个子目录
│       ├── qrcode/                   # 商家二维码
│       │   └── store_qrcode.png
│       ├── cover/                    # 店铺封面图
│       │   └── cover_xxx.jpg
│       └── products/                 # 菜谱/商品图片
│           ├── prod_{productId}_1.jpg
│           └── prod_{productId}_2.jpg
├── categories/                       # 分类图标
│   └── cat_{categoryId}.png
└── temp/                             # 临时上传目录（定期清理）
    └── tmp_xxx.jpg
```

### 8.3 文件约束

| 约束项 | 规则 |
|--------|------|
| 允许格式 | jpg、jpeg、png、webp |
| 最大文件大小 | 5MB |
| 图片处理 | 上传时自动压缩，最大宽度 1200px |
| 文件命名 | `{类型前缀}_{业务ID}_{序号}.{扩展名}` |
| 访问路径 | `/api/files/{相对路径}`（后端静态文件中间件） |
| 清理策略 | temp 目录超过 24 小时的文件自动清理 |

### 8.4 图片上传交互

系统中所有图片相关功能均需支持"选择文件上传"，不能只要求用户手动填写图片 URL。URL 字段可作为兼容或高级输入保留，但默认入口应为文件选择。

| 场景 | 上传入口 | 结果写入 |
|------|---------|---------|
| 店铺封面图 | 商家端店铺管理页选择本地图片文件 | `Store.coverImageUrl` |
| 商品图片 | 商家端商品新增/编辑页选择本地图片文件 | `Product.imageUrl` |
| 套餐图片 | 商家端套餐新增/编辑页选择本地图片文件 | 套餐商品的图片字段 |
| 分类图标 | 商家端分类新增/编辑页选择本地图标文件，或选择预设图标 | `Category.icon` |
| 模板封面图 | 管理端模板新增/编辑页选择本地图片文件 | `StoreTemplate.coverImageUrl` |
| 模板商品/套餐图片 | 管理端模板商品维护页选择本地图片文件 | 模板商品或模板套餐图片字段 |

**上传流程要求：**

```
用户点击上传/更换图片
  → 浏览器打开本地文件选择器
  → 前端校验格式、大小并展示本地预览
  → 使用 multipart/form-data 上传文件
  → 后端保存到对应业务目录并返回图片访问 URL
  → 前端将 URL 写入当前表单
  → 用户保存业务表单后，图片与店铺/商品/分类/模板正式绑定
```

| 要求 | 说明 |
|------|------|
| 上传方式 | 使用 `multipart/form-data`，字段名统一为 `file`，同时传入上传场景（如 store_cover/product/category_icon/template）和业务 ID |
| 上传反馈 | 上传中显示进度或加载状态，成功后立即展示预览，失败时保留原图片并提示原因 |
| 替换图片 | 替换成功后新图片生效，旧图片进入待清理或引用检查流程，避免误删仍被其他模板/商品引用的图片 |
| 删除图片 | 支持清空可选图片字段；删除只解除业务引用，物理文件由后台清理策略处理 |
| 权限校验 | 商家只能上传和绑定自己店铺下的图片；管理员可上传模板和平台资源图片 |
| 移动端体验 | H5 页面需使用系统原生文件选择能力，支持从相册或拍照选择图片（取决于浏览器能力） |
| 图片一致性 | 业务表单保存失败时，已上传但未绑定的图片保留在 temp 目录，按清理策略自动删除 |

## 9. 非功能性需求

### 9.1 性能

- H5 首屏加载 ≤ 2 秒（4G 网络）
- 商品列表滚动流畅，支持懒加载
- 订单状态推送延迟 ≤ 3 秒
- 图片加载使用缩略图 + 懒加载

### 9.2 可用性

- H5 页面适配主流手机浏览器（微信内置浏览器、Safari、Chrome）
- 商家端 H5 手机端优先适配，PC 端可用
- 弱网环境下可浏览已加载内容
- 后期商家端可改造为微信小程序

### 9.3 安全

- 商家端接口需 JWT 认证
- 顾客端无需登录，通过取货码标识订单
- 防止恶意刷单（频率限制：同一 IP 每分钟最多 5 单）
- 商品价格以后端为准，前端不可篡改
- 文件上传需校验文件类型和大小

### 9.4 数据

- 订单数据保留至少 1 年
- 支持数据导出（CSV）

### 9.5 测试与验收

- 核心业务服务需有单元测试覆盖，重点覆盖注册登录、店铺搜索、店铺管理、商品分类、下单、取货码、订单消息、优惠、后台管理和经营数据
- 顾客扫码短链 `/A/{storeCode}`、菜单加载、购物车、下单、订单状态流转需纳入集成测试
- API 集成测试统一校验响应包装 `{ code, message, data }`
- 测试数据需要可重复生成，不依赖固定手机号、固定订单号等易冲突数据
- 已知缺陷修复后需补充回归测试说明，避免同类问题再次出现
- 演示环境至少包含一套默认商家、店铺码、菜品、优惠和订单数据，便于人工验收

## 10. 技术选型确认

| 层 | 技术 | 说明 |
|---|---|---|
| 顾客端 | Vue 3 + Vite (H5) | 手机端适配，响应式布局 |
| 商家端 | Vue 3 + Vite (H5) | 手机端优先适配，后期改造为微信小程序 |

## 11. 优先级与分期

### 第一期（MVP）

- 顾客端：短链扫码入店（`/A/{storeCode}`）、店铺搜索、菜单浏览（含规格选择）、购物车、下单（含 6 位 Base36 取货码生成和二维码展示）、订单跟踪、顾客直接注册/登录、我的订单按店铺分组、订单消息会话
- 商家端：注册登录、统一登录、店铺管理、店铺码/二维码/分享、分类管理、商品管理（含规格）、订单管理（含取货码输入核销、二维码扫码核销、实时提醒）、顾客消息会话处理
- 平台管理端：行业分类管理、商家模板管理、默认管理员账号初始化
- 系统级：SignalR 实时推送与消息通讯、本地文件存储、商家模板系统、匿名设备标识、统一 API 响应、基础测试覆盖

### 第二期

- 顾客端：商品搜索、再来一单、店铺收藏/常用店铺
- 商家端：经营数据、满减活动
- 平台管理端：商家审核/禁用/解禁、审计日志与操作日志
- 系统级：短信通知、更多集成测试和数据导出验收

### 第三期

- 商家端：店员管理、折扣活动、活动叠加规则
- 系统级：小票打印、多店铺
- 在线支付（微信支付）
- 商家端改造为微信小程序
