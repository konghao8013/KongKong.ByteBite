# 空空码上点单（KongKong.ByteBite）— 需求清单

## 1. 产品定位

**空空码上点单**（KongKong.ByteBite）是面向烧烤店、小卖部、饭馆的轻量级扫码点单系统。顾客通过扫描店铺二维码进入 H5 页面浏览菜单并下单，商家通过 H5 管理后台管理店铺、菜单和订单。后期商家端可改造为微信小程序。

## 2. 用户角色

| 角色 | 说明 | 使用端 |
|------|------|--------|
| 顾客 | 扫码进入店铺，浏览菜单/商品，下单 | H5 网页（手机浏览器） |
| 商家/店员 | 管理店铺、菜单/商品、接单、处理订单 | H5 网页（手机端优先适配，后期改造为微信小程序） |
| 平台管理员 | 管理商家入驻、行业模板管理、平台配置 | Web 后台（PC） |

## 3. 核心业务流程

```
商家创建店铺 → 选择行业模板（可选）→ 一键应用模板商品 → 微调价格 → 生成店铺二维码 → 贴码到店铺
顾客扫码 → 进入店铺 H5 → 浏览菜单 → 加入购物车 → 提交订单 → 获得取货码
商家收到新订单提醒 → 接单 → 制作/备货 → 通知顾客取餐 → 顾客凭取货码取餐 → 完成订单
顾客线下付款 → 订单关闭
```

## 4. 功能模块

### 4.1 顾客端（H5 网页）

#### 4.1.1 扫码入店 [P0]

- 顾客扫描店铺二维码，进入对应店铺的 H5 点单页面
- 无需登录、无需手机号即可浏览菜单和下单
- 浏览器缓存店铺访问记录，下次进入自动跳转或展示最近店铺

#### 4.1.2 会话恢复 [P0]

顾客再次扫码进入时，自动恢复上次的操作状态，实现无缝衔接：

| 场景 | 恢复行为 | 存储位置 |
|------|---------|---------|
| 有未提交的购物车 | 自动恢复购物车内容（商品、规格、数量、备注） | localStorage |
| 有进行中的订单 | 直接跳转到订单跟踪页，展示取货码和实时状态 | localStorage + 服务端 |
| 无购物车无进行中订单 | 进入菜单浏览页，定位到上次浏览的分类 | localStorage |
| 换了店铺扫码 | 清除旧店铺缓存，进入新店铺首页 | localStorage |

```json

          "productId": "uuid",
          "productName": "烤羊肉串",
          "price": 15,
          "quantity": 2,
          "specs": [{ "groupId": "uuid", "optionId": "uuid", "name": "大份", "extraPrice": 10 }],

```

缓存策略：

| 策略 | 规则 |
|------|------|
| 购物车过期 | 同一店铺购物车缓存 24 小时有效，超时清空 |
| 订单历史上限 | 保留最近 20 条已完成订单的取货码 |
| 进行中订单 | 订单完成后 24 小时从活跃列表移至历史 |
| 换店铺 | 扫不同店铺的码时，清空当前购物车（提示用户确认），保留历史订单 |
| 缓存冲突 | 服务端数据为准，本地缓存仅做恢复提示，不覆盖服务端状态 |

#### 4.1.3 菜单浏览 [P0]

- 按分类浏览菜品/商品列表
- 每个商品展示：图片、名称、价格（起售价，多规格时显示"起"）、描述、销量
- 商品售罄/下架状态展示（灰色标记）
- 分类左侧导航 + 右侧商品列表布局

**商家营业状态对顾客端的影响：**

| 商家状态 | 顾客端展示 | 能否下单 |
|---------|-----------|---------|
| 营业中 | 正常展示菜单，可浏览可下单 | ✅ 可以 |
| 休息中 | 顶部横幅提示"商家休息中，暂不接单"，菜单仍可浏览但按钮置灰 | ❌ 不可以 |
| 营业时间外 | 顶部横幅提示"商家尚未营业（营业时间 XX:XX-XX:XX）"，可浏览菜单 | ❌ 不可以 |

- 休息中/营业时间外，顾客仍可浏览菜单和添加购物车，但"去结算"按钮置灰
- 购物车内容保留，商家恢复营业后顾客可直接下单
- 商家切换营业状态时，已连接的顾客端通过 SignalR 实时更新页面状态

#### 4.1.4 商品搜索 [P1]

- 关键词搜索菜品/商品
- 搜索结果高亮匹配文字

#### 4.1.5 购物车 [P0]

- 添加/删除/修改商品数量
- 选择商品规格（如：大份/小份，不同规格不同价格）
- 单品备注（少辣、不加香菜等）
- 实时计算合计金额
- 购物车浮动按钮（显示商品数量角标）
- 清空购物车

#### 4.1.6 下单 [P0]

- 确认订单信息（商品列表、规格、数量、备注、合计）
- 选择就餐方式：堂食 / 打包 / 外卖
- 堂食时填写桌号（手动输入，非扫码绑定）
- 外卖时填写配送地址和联系电话
- 提交订单（先做后付，无需在线支付）
- 下单成功后生成取货码（数字/字母组合，4-6位）
- 取货码页面展示，支持截图保存

**优惠计算流程（下单确认页）：**

```
购物车合计 → 系统自动匹配可用优惠活动
         → 展示优惠信息：
             ├── 满减：满50减5（当前合计¥66，已满足）
             ├── 折扣：烧烤类8折（当前合计中烧烤类¥50，折后¥40）
             └── 优惠不叠加提示（商家配置）
         → 顾客选择优惠方式（若可叠加则自动应用最优组合）
         → 实时更新：
             ├── 商品合计：¥66.00
             ├── 优惠金额：-¥5.00（满减）
             ├── 应付金额：¥61.00
             └── 优惠明细展开/收起
         → 提交订单
```

**下单确认页展示元素：**

| 区域 | 内容 |
|------|------|
| 商品列表 | 每个商品的名称、规格、数量、单价、小计 |
| 优惠区域 | 可用满减/折扣列表，自动推荐最优，顾客可手动切换 |
| 优惠明细 | 优惠类型、优惠金额、满足条件说明 |
| 合计区域 | 商品合计、优惠减免、应付金额 |
| 就餐方式 | 堂食（填桌号）/ 打包 / 外卖（填地址） |
| 备注 | 整单备注（如：不要葱） |

**优惠匹配规则：**

| 规则 | 说明 |
|------|------|
| 自动匹配 | 系统自动计算所有可用优惠，推荐最优组合 |
| 顾客可切换 | 顾客可在多个可用优惠中手动选择 |
| 不满足时提示 | "再买 ¥XX 可享受满减优惠"，引导凑单 |
| 优惠互斥 | 商家配置是否允许满减与折扣叠加 |
| 价格以后端为准 | 前端展示为参考，提交时后端重新计算并锁定 |

#### 4.1.7 取货码与订单跟踪 [P0]

- 下单成功后展示取货码
- 取货码存储在浏览器缓存（localStorage）
- 再次进入时可通过取货码查看订单状态
- 订单状态：待接单 → 制作中 → 待取餐 → 已完成
- 订单状态变更时页面实时更新（SignalR）

**顾客取消订单：**

| 订单状态 | 能否取消 | 说明 |
|---------|---------|------|
| 待接单 | ✅ 可以 | 顾客可随时取消，订单直接标记"已取消" |
| 制作中 | ❌ 不可以 | 商家已开始制作，顾客需联系商家协商 |
| 待取餐 | ❌ 不可以 | 商家已完成制作，顾客需到店取餐 |
| 已完成 | ❌ 不可以 | 订单已关闭 |

- 顾客在"待接单"状态下点击"取消订单"→ 二次确认 → 订单标记"已取消"
- 取消后取货码失效，从活跃订单列表移除
- 商家端收到订单取消通知（SignalR）

**商家拒单：**

- 商家在"待接单"状态下可拒单，需填写拒单原因
- 顾客端实时收到拒单通知，展示拒单原因
- 取货码失效，订单标记"已取消"

**退款流程（后期）：**

- MVP 阶段为"先做后付"，无需退款流程
- 后期接入在线支付后，取消/拒单自动退款

#### 4.1.8 订单历史 [P1]

- 浏览器缓存记录历史取货码列表
- 点击取货码查看对应订单详情
- 再来一单（一键加入购物车）

#### 4.1.9 顾客注册/登录 [P1]

顾客可选择性注册账号，注册后将浏览器缓存中的匿名数据合并到账号中，实现跨设备数据同步。

**注册/登录方式：**

| 阶段 | 方式 | 说明 |
|------|------|------|
| 第一期 | 手机号 + 短信验证码 | 轻量注册，无需密码 |
| 后期 | 微信授权登录 | 小程序版使用 |

**注册流程与数据合并：**

```
顾客扫码点单（匿名）→ 浏览器缓存积累数据（订单、购物车、访问记录）
                  → 选择注册
                  → 输入手机号 + 验证码
                  → 系统检测到本地缓存数据
                  → 提示"发现您之前在以下店铺有订单记录，是否合并到新账号？"
                  → 展示待合并数据摘要：
                      ├── XX烧烤店：3笔订单（1笔进行中、2笔已完成）
                      ├── XX便利店：1笔订单（已完成）
                      └── XX面馆：购物车中有2件商品
                  → 用户确认合并
                  → 系统执行合并：
                      1. 匿名订单 → 关联到注册用户
                      2. 浏览器缓存购物车 → 同步到用户账号
                      3. 访问记录 → 同步到用户账号
                  → 合并完成，清除本地缓存
                  → 后续登录自动同步数据
```

**多店铺数据合并规则：**

| 数据类型 | 合并策略 | 说明 |
|----------|---------|------|
| 进行中订单 | 全部合并 | 按店铺分组展示，保留取货码 |
| 已完成订单 | 全部合并 | 合并到账号订单历史，按时间排序 |
| 购物车 | 按店铺分别合并 | 保留各店铺购物车内容，不跨店混合 |
| 访问记录 | 合并去重 | 保留最近访问的店铺列表 |
| 取货码 | 全部合并 | 合并到账号取货码列表 |

**数据合并冲突处理：**

| 场景 | 处理方式 |
|------|---------|
| 本地缓存与账号已有相同店铺订单 | 以服务端数据为准，本地数据补充缺失的订单 |
| 本地购物车与账号购物车冲突 | 合并去重，同一商品取数量较大值 |
| 同一订单在本地和账号都存在 | 关联到账号，不重复创建 |

**注册后数据同步：**

| 场景 | 行为 |
|------|------|
| 已登录用户扫码 | 订单直接关联到账号，不存 localStorage |
| 已登录用户换设备 | 登录后自动拉取账号数据，无需手动合并 |
| 已登录用户退出登录 | 降级为匿名模式，数据存 localStorage |
| 匿名用户在已注册设备注册 | 自动触发合并流程 |

**顾客账号信息：**

| 字段 | 必填 | 说明 |
|------|------|------|
| 手机号 | 是 | 作为账号标识，唯一 |
| 昵称 | 否 | 默认为手机号后四位 |
| 头像 | 否 | 默认系统头像 |

#### 4.1.10 顾客退出登录 [P1]

已登录的顾客可主动退出登录，退出后降级为匿名模式继续使用。

**退出登录行为：**

| 行为 | 说明 |
|------|------|
| 清除登录态 | 前端清除 Token/用户信息缓存 |
| 降级为匿名模式 | 退出后仍可扫码点单，数据存 localStorage |
| 进行中订单保留 | 退出登录不影响已有订单，取货码仍可查看 |
| 购物车保留 | 退出登录后购物车内容不清空 |
| API 接口 | `POST /api/customers/logout`，服务端清除 Token 即可 |

### 4.2 商家端（H5 网页，后期改造为微信小程序）

#### 4.2.1 商家注册/登录 [P0]

**注册流程：**

| 阶段 | 注册方式 | 审核要求 | 注册后状态 | 说明 |
|------|---------|---------|-----------|------|
| 第一期（MVP） | 手机号 + 密码 | 无需审核 | 直接激活，可立即使用 | 降低门槛，快速获客 |
| 后期 | 手机号 + 密码 | 需平台管理员审核 | 待审核状态，无法使用 | 控制商家质量 |

**注册信息：**

| 字段 | 必填 | 说明 |
|------|------|------|
| 手机号 | 是 | 作为登录账号，唯一 |
| 密码 | 是 | 最少 6 位，BCrypt 加密存储 |
| 店铺名称 | 是 | 注册时同时创建店铺 |
| 行业类型 | 否 | 可选择行业分类（关联模板系统），也可跳过 |
| 验证码 | 是 | 手机短信验证码（第一期可降级为图形验证码） |

**商家账号状态：**

| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| 待审核 | 后期注册后等待管理员审核 | 仅可查看，无法管理店铺 |
| 已激活 | 审核通过 / MVP 直接激活 | 全部功能 |
| 已禁用 | 管理员手动禁用（违规等） | 无法登录 |

**登录方式：**

| 阶段 | 登录方式 |
|------|---------|
| 第一期 | 手机号 + 密码 |
| 后期 | 手机号 + 密码 / 微信授权登录（绑定手机号） |

**后期审核功能（平台管理员）：**

- 待审核商家列表
- 审核操作：通过 / 拒绝（需填写原因）
- 批量审核
- 商家禁用/解禁
- 审核通知：短信通知商家审核结果

#### 4.2.2 店铺管理 [P0]

- 店铺名称、描述、封面图
- 营业状态切换（营业中 / 休息中）
- 店铺二维码生成与下载
- 营业时间设置
- **配送设置**：
  - 就餐方式开关：堂食 / 打包 / 外卖（商家可分别启用/禁用）
  - 外卖最低消费：商家可设置外卖订单的最低金额（如：外卖满 ¥30 起送）
  - 打包费：商家可设置打包费（如：每单 ¥1）
  - 顾客端下单时，若选择外卖且金额未达最低消费，提示"外卖需满 ¥XX 起送"
  - 堂食和打包不设最低消费

#### 4.2.3 分类管理 [P0]

商家可完全自定义商品分类，不受行业模板限制，自由创建符合自身经营特色的分类体系。

**分类属性：**

| 属性 | 说明 | 示例 |
|------|------|------|
| 分类名称 | 自定义，最多 20 字 | 进店福利、新疆烧烤、鸡和海鲜类 |
| 分类图标 | 可选，上传图标或选择预设图标 | 🎁 🍗 🦐 🥬 🍱 🔥 🍺 |
| 分类类型 | 普通分类 / 特殊分类 | 见下方分类类型说明 |
| 排序权重 | 数值越大越靠前 | 热销=1000，进店福利=900 |
| 显示状态 | 显示 / 隐藏 | 临时下架某分类时使用 |

**分类类型：**

| 类型 | 说明 | 顾客端展示 | 商品来源 |
|------|------|-----------|---------|
| 普通分类 | 商家手动归类的商品 | 正常展示，按排序权重排列 | 商家手动将商品归入 |
| 热销 | 自动聚合销量 Top N 的商品 | 置顶展示，标签"🔥 热销" | 系统自动按销量计算，商家可设置 Top 数量（默认 10） |
| 进店福利 | 优惠/引流商品，吸引下单 | 置顶展示，标签"🎁 进店福利" | 商家手动将商品归入 |
| 套餐 | 组合商品，多件打包优惠 | 正常展示，标签"🍱 套餐" | 商家手动创建套餐商品 |

**分类管理操作：**

- 添加分类：输入名称、选择类型、上传图标（可选）、设置排序
- 编辑分类：修改名称、图标、类型、排序
- 删除分类：分类下有商品时需先移走商品或选择一并删除
- 拖拽排序：支持拖拽调整分类显示顺序
- 批量操作：批量隐藏/显示分类

**分类与商品关系：**

| 规则 | 说明 |
|------|------|
| 一个商品只能属于一个普通分类 | 如"烤羊肉串"属于"新疆烧烤"分类 |
| 一个商品可同时出现在特殊分类 | 如"烤羊肉串"属于"新疆烧烤"，同时因销量高自动出现在"热销" |
| 热销分类自动维护 | 系统按近 7 天销量自动计算，商家无需手动管理 |
| 套餐分类下的商品为虚拟商品 | 套餐商品由多个普通商品组合而成，有独立的组合价格 |
| 分类为空时顾客端不展示 | 某分类下无上架商品时，顾客端自动隐藏该分类 |

**烧烤店分类示例：**

| 排序 | 分类名称 | 类型 | 说明 |
|------|---------|------|------|
| 1 | 🔥 热销 | 热销 | 自动聚合销量 Top 10 |
| 2 | 🎁 进店福利 | 进店福利 | 凉菜、小食等引流品 |

### 4.3 系统级功能

#### 4.3.1 SignalR 实时推送 [P0]

SignalR 是本系统实时通信的核心，顾客端和商家端均需接入，确保订单状态变更即时触达。

**推送场景与消息定义：**

| 编号 | 推送方向 | 触发事件 | 消息内容 | 接收端行为 |
|------|---------|---------|---------|-----------|
| S1 | 顾客 → 商家 | 顾客下单 | 订单摘要（订单号、商品数、合计、就餐方式） | 商家端弹窗提醒 + 声音 + 震动，订单列表刷新 |
| S2 | 商家 → 顾客 | 商家接单 | 接单确认（预计等待时间，可选） | 顾客端订单状态更新为"制作中"，页面提示 |
| S3 | 商家 → 顾客 | 商家拒单 | 拒单原因 | 顾客端订单状态更新为"已取消"，展示拒单原因 |
| S4 | 商家 → 顾客 | 备餐完毕/待取餐 | 取餐提醒 + 取货码 | 顾客端订单状态更新为"待取餐"，高亮取货码 |
| S5 | 商家 → 顾客 | 外卖已送出 | 送出提醒 + 预计送达时间（可选） | 顾客端订单状态更新为"配送中"（预留） |
| S6 | 商家 → 顾客 | 订单完成 | 完成确认 | 顾客端订单状态更新为"已完成" |
| S7 | 系统 → 商家 | 订单超时未接 | 超时提醒（订单号 + 等待时长） | 商家端提醒该订单已等待超过 N 分钟 |

**连接管理：**

| 环节 | 顾客端 | 商家端 |
|------|--------|--------|
| 连接时机 | 进入店铺页面时自动建立连接 | 登录后自动建立连接，全程保持 |
| 连接标识 | 匿名连接，通过订单 ID 订阅特定订单消息 | 认证连接，通过店铺 ID 订阅店铺所有订单消息 |
| 断线重连 | 自动重连（指数退避，最多 5 次） | 自动重连（指数退避，最多 10 次） |
| 重连后 | 主动拉取订阅订单的最新状态，补齐断线期间变更 | 主动拉取店铺订单列表最新状态 |
| 离开页面 | 断开连接，释放服务端资源 | 保持连接（多标签页共享连接） |

**Hub 设计：**

```
SignalR Hubs
├── OrderHub（顾客端）
│   ├── 订阅订单：SubscribeOrder(orderId)
│   ├── 取消订阅：UnsubscribeOrder(orderId)
│   └── 服务端推送方法：
│       ├── OrderStatusChanged(orderId, status, data)
│       └── OrderCancelled(orderId, reason)
│
└── StoreHub（商家端）
    ├── 订阅店铺：SubscribeStore(storeId)
    ├── 取消订阅：UnsubscribeStore(storeId)
    └── 服务端推送方法：
        ├── NewOrderReceived(orderId, summary)
        └── OrderTimeoutWarning(orderId, waitingMinutes)
```

**降级策略：**

| 场景 | 降级行为 |
|------|---------|
| SignalR 连接失败 | 降级为 HTTP 轮询（每 5 秒查询一次订单状态） |
| 轮询也失败 | 页面展示"网络异常"提示，提供手动刷新按钮 |
| 商家端长时间无响应 | 顾客端展示"商家可能未在线，请耐心等待或到店咨询" |


**多商家组合流程：**

```

  → 管理员调整：
      ├── 合并同名分类（如：商家A和商家B都有"烧烤"分类 → 合并为一个分类）

      └── 修改参考价格
  → 保存模板
```

**从商家引入的规则：**

| 规则 | 说明 |
|------|------|
| 价格转为参考价 | 商家实际售价转为模板"参考价格"，新商家应用后可自行调整 |
| 图片复制 | 商家商品图片复制到模板资源目录，不引用原商家图片（避免商家删图影响模板） |
| 不影响原商家 | 引入操作只读取商家数据，不修改原商家的任何内容 |
| 商家无感知 | 商家不知道自己的商品被引入模板（无需商家授权） |
| 同名分类合并 | 多商家组合时，同名分类自动合并，商品归入同一分类 |
| 同名商品处理 | 多商家组合时，同名商品默认保留第一个引入的，管理员可选择保留哪个或都保留 |
| 套餐可引入 | 商家的套餐商品也可引入模板，子商品引用自动映射为模板商品引用 |
| 引入后独立 | 引入后的模板数据与原商家完全独立，后续商家修改不影响模板 |

**关键规则：**

| 规则 | 说明 |
|------|------|
| 模板商品默认下架 | 应用后商品状态为"下架"，商家确认价格后才上架，避免错误价格暴露给顾客 |
| 价格仅供参考 | 模板价格为市场参考价，商家必须自行确认后才能上架 |
| 图片可替换 | 模板图片为通用示意图，商家应用后可替换为自己的实拍图 |
| 可跳过模板 | 商家可选择"自定义"跳过模板，从零配置 |
| 可部分应用 | 商家可只应用模板中的部分分类和商品，不需要全部采用 |
| 模板不锁定 | 应用模板后，商家可自由增删改分类和商品，不受模板约束 |
| 可重复应用 | 商家可在已有商品的基础上，追加应用其他模板（合并模式） |

## 5. 取货码机制

### 5.1 设计说明

取货码是本系统的核心交互凭证，替代传统手机号验证流程：

| 环节 | 说明 |
|------|------|
| 生成时机 | 顾客下单成功后自动生成 |
| 编码规则 | 4-6 位大写字母 + 数字组合（排除易混淆字符：0/O、1/I/L） |
| 唯一性 | 同一店铺内唯一，跨店铺可重复 |
| 存储方式 | 浏览器 localStorage 缓存取货码列表 |
| 展示方式 | 订单详情页大字展示，支持截图 |
| 核销方式 | 商家端输入取货码或扫码核销 |
| 有效期 | 订单完成后保留 24 小时，之后从活跃列表移除 |

### 5.2 顾客端取货码流程

```
下单成功 → 展示取货码页面（大字 + 倒计时提示）
         → 取货码存入 localStorage
         → 再次进入 → 自动展示未完成订单的取货码列表
         → 点击取货码 → 查看订单详情与状态
```

### 5.3 商家端取货码核销

```
订单标记待取餐 → 顾客出示取货码
              → 商家输入取货码 → 系统匹配订单
              → 确认核销 → 订单标记已完成
```

## 6. 商品规格设计

### 6.1 数据模型

```
商品（Product）
├── 基础价格（默认规格价格）
├── 规格组（SpecGroup）
│   ├── 规格组名称（如：份量）
│   └── 规格项（SpecOption）
│       ├── 规格名称（如：小份）
│       ├── 加价（如：+0 / +10）
│       └── 库存（可选）
└── 规格组（SpecGroup）
    ├── 规格组名称（如：辣度）
    └── 规格项（SpecOption）
        ├── 规格名称（如：微辣）
        ├── 加价（如：+0）
        └── 库存（可选）
```

### 6.2 价格计算

- 商品最终价格 = 基础价格 + 各规格组选中项的加价之和
- 顾客端显示"起售价"（基础价格），选择规格后实时更新价格
- 购物车和订单中记录每个商品的具体规格和最终价格

### 6.3 示例

| 商品 | 基础价格 | 规格组 | 规格项 | 加价 | 最终价格 |
|------|---------|--------|--------|------|---------|
| 烤羊肉串 | ¥15 | 份量 | 小份 | +0 | ¥15 |
| | | | 大份 | +10 | ¥25 |
| | | 辣度 | 不辣 | +0 | — |
| | | | 微辣 | +0 | — |
| | | | 特辣 | +0 | — |
| 可乐 | ¥3 | 规格 | 听装 | +0 | ¥3 |
| | | | 瓶装 | +2 | ¥5 |

### 6.4 套餐商品数据模型

套餐是一种特殊的虚拟商品，由多个普通商品组合而成，拥有独立的组合价格。

```
套餐商品（ComboProduct）
├── 基础信息
│   ├── 套餐名称（如：2人超值套餐）
│   ├── 套餐价格（独立定价，不等于子商品价格之和）
│   ├── 套餐描述（如：含烤羊肉串2份+烤鸡翅4份+啤酒2瓶）
│   ├── 套餐图片
│   ├── 所属分类（套餐类型分类）
│   ├── 排序权重
│   ├── 商品状态：上架 / 下架 / 售罄
│   ├── 起购数量（默认 1）
│   └── 可选规格组（如：辣度选择，套餐整体适用）
│
├── 包含商品列表（ComboItem）
│   └── 子商品引用
│       ├── 引用的普通商品 ID
│       ├── 数量（如：2 份烤羊肉串）
│       ├── 默认规格（如：大份）
│       ├── 是否允许顾客替换规格（可选）
│       └── 子商品备注（如：默认微辣）
│
└── 套餐优惠信息
    ├── 原价总和（各子商品按默认规格的价格之和）
    ├── 套餐价格（独立定价，通常低于原价总和）
    └── 优惠金额（原价总和 - 套餐价格）
```

**套餐价格计算规则：**

| 规则 | 说明 |
|------|------|
| 套餐独立定价 | 套餐价格由商家手动设置，不自动计算 |
| 顾客端展示 | 显示"原价 ¥120 · 套餐价 ¥88 · 省 ¥32" |
| 子商品规格替换 | 若顾客替换了子商品规格，套餐价格不变（规格差价另计） |
| 套餐与满减 | 套餐商品可参与满减活动，套餐价格计入满减门槛 |
| 套餐内商品售罄 | 任一子商品售罄，整个套餐自动标记售罄 |
| 套餐内商品下架 | 任一子商品下架，整个套餐自动下架 |

**套餐示例：**

| 套餐 | 套餐价 | 包含商品 | 原价总和 | 优惠金额 |
|------|--------|---------|---------|---------|
| 2人超值套餐 | ¥88 | 烤羊肉串×2(大份) + 烤鸡翅×4 + 烤茄子×1 + 雪花啤酒×2(瓶装) | ¥120 | ¥32 |
| 4人畅吃套餐 | ¥168 | 烤羊肉串×4(大份) + 烤鱼×1(整条) + 烤鸡翅×8 + 烤韭菜×2 + 雪花啤酒×4(瓶装) | ¥236 | ¥68 |

## 7. 优惠活动设计

### 7.1 满减活动

```
满减规则（DiscountRule）
├── 活动名称（如：满50减5）
├── 满足金额（如：50）
├── 减免金额（如：5）
├── 适用范围：全店 / 指定分类
├── 有效期：开始时间 ~ 结束时间
└── 状态：启用 / 停用
```

- 支持多档满减（如满50减5、满100减15），自动匹配最高档
- 顾客端下单页实时展示可用满减
- 订单记录实际优惠金额

### 7.2 折扣活动

```
折扣规则（DiscountRule）
├── 活动名称（如：烧烤类8折）
├── 折扣率（如：0.8 = 8折）
├── 适用范围：全店 / 指定分类 / 指定商品
├── 有效期：开始时间 ~ 结束时间
└── 状态：启用 / 停用
```

- 折扣与满减可叠加（商家可配置是否允许叠加）
- 折扣金额四舍五入到分

### 7.3 优先级

- MVP 先做满减活动
- 折扣活动第二期
- 叠加规则第三期细化

## 8. 文件存储设计

### 8.1 存储方式

图片直接存储在 .NET 后端的资源目录下，不使用 OSS/COS。

### 8.2 目录结构

```
uploads/
├── stores/                           # 店铺资源
│   └── {storeId}/                    # 每个商家一个子目录
│       ├── qrcode/                   # 商家二维码
│       │   └── store_qrcode.png
│       ├── cover/                    # 店铺封面图
│       │   └── cover_xxx.jpg
│       └── products/                 # 菜谱/商品图片
│           ├── prod_{productId}_1.jpg
│           └── prod_{productId}_2.jpg
├── categories/                       # 分类图标
│   └── cat_{categoryId}.png
└── temp/                             # 临时上传目录（定期清理）
    └── tmp_xxx.jpg
```

### 8.3 文件约束

| 约束项 | 规则 |
|--------|------|
| 允许格式 | jpg、jpeg、png、webp |
| 最大文件大小 | 5MB |
| 图片处理 | 上传时自动压缩，最大宽度 1200px |
| 文件命名 | `{类型前缀}_{业务ID}_{序号}.{扩展名}` |
| 访问路径 | `/api/files/{相对路径}`（后端静态文件中间件） |
| 清理策略 | temp 目录超过 24 小时的文件自动清理 |

## 9. 非功能性需求

### 9.1 性能

- H5 首屏加载 ≤ 2 秒（4G 网络）
- 商品列表滚动流畅，支持懒加载
- 订单状态推送延迟 ≤ 3 秒
- 图片加载使用缩略图 + 懒加载

### 9.2 可用性

- H5 页面适配主流手机浏览器（微信内置浏览器、Safari、Chrome）
- 商家端 H5 手机端优先适配，PC 端可用
- 弱网环境下可浏览已加载内容
- 后期商家端可改造为微信小程序

### 9.3 安全

- 商家端接口需 JWT 认证
- 顾客端无需登录，通过取货码标识订单
- 防止恶意刷单（频率限制：同一 IP 每分钟最多 5 单）
- 商品价格以后端为准，前端不可篡改
- 文件上传需校验文件类型和大小

### 9.4 数据

- 订单数据保留至少 1 年
- 支持数据导出（CSV）
